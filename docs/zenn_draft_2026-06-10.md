## 1. 背景と問い: Agentic AI 時代の「再利用」設計

LLM エージェント（Agentic AI / AI エージェント）を「タスクごとにゼロから組む」のは無駄が大きい。
特に同じドメイン（例: NDA リーガルレビュー、業務委託契約レビュー、利用規約チェック等の **リーガルテック**）の中で類似タスクが並ぶとき、
**何を再利用できて、何を再利用してはいけないのか** を明確にしたい。

最近の Agentic AI 系研究（Voyager、AgentSquare、AFlow、ADAS、Skill-Pro 等）の整理から、
エージェントは大きく 3 層に分けて議論されることが多い:

| 層 | 内容 | 再利用可能性（先行研究の通説） |
| --- | --- | --- |
| 知識層 (Knowledge) | ドメイン辞書、ナレッジベース、規制要件 | 高 |
| ツール層 (Tools) | API、DB アクセス、ファイル操作 | 高（入出力契約が同じ範囲で） |
| ポリシー層 (Policy) | プロンプト、推論手順、モデル選択 | 低（タスクごとに最適化が必要） |

参考: Voyager（スキルライブラリ）、AgentSquare（モジュール探索）、AFlow（ワークフロー探索）、DSPy（プロンプト最適化）など。

**未検証だったこと**: 知識層再利用が **実務ドメイン** で **本当に効くのか**、そして **モデル能力に依存しないのか**。
本稿の検証は、ここに 1 つの実測データを足すことを目的とする。

## 2. 中核仮説

> 同一ドメイン内の関連タスク T1, T2 において、
> T1 で構築した知識層（ドメイン辞書）を T2 に注入すると、
> T2 をゼロベース構築するより **コールドスタート工数を削減し、最終スコアを劣化させず、負の転移を起こさない**。

## 3. 検証設計

### 3.1 タスク選定（NDA ドメイン）

| | 内容 |
| --- | --- |
| T1: NG 条項検出 | 与えられた契約条項に、辞書定義された NG パターンが含まれるかを列挙 |
| T2: NG 条項修正 | NG が含まれる条項を最小変更でリライト |
| 関係 | T1 で「何が NG か」が定義されていれば、T2 で「どう直すか」に再利用できるはず |

NDA に絞った理由: 法務ドメインで実務価値が高く、NG パターンが既知（経済産業省「秘密情報の保護ハンドブック」等）であり、ラベル付き評価器を作りやすい。

### 3.2 知識層（NG パターン辞書）

中小企業庁 NDA 雛形を元に、9 種類の NG パターンを定義:

| ID | 名前 | 型 |
| --- | --- | --- |
| nda_scope_overbroad | 秘密情報の範囲過大 | 明示型 |
| nda_duration_unbounded | 秘密保持期間の無期限・過長 | 明示型 |
| nda_purpose_undefined | 利用目的の不明確 | 明示型 |
| nda_disclosure_exception_missing | 法令・規制当局等への開示例外不在 | 欠落型 |
| nda_remedy_imbalanced | 損害賠償・違約金の不均衡 | 明示型 |
| nda_jurisdiction_one_sided | 準拠法・管轄の一方的指定 | 明示型 |
| nda_return_destroy_missing | 返還・廃棄義務の不在 | 欠落型 |
| nda_derivative_undefined | 派生情報・成果物の取扱い未定義 | 欠落型 |
| nda_survival_missing | 存続条項の欠落 | 欠落型 |

YAML 辞書として SemVer 管理し、各パターンには「検出すべき / 紛らわしい / 対象条項」の 3 セクション記述。

### 3.3 評価設計

| 指標 | 用途 |
| --- | --- |
| NG Recall（主指標） | 業務上の致命傷は「NG を OK と誤判定」する見逃しのため |
| Precision（補助） | 全件 NG 判定への偏りを抑える |
| macro_F2（β=2、Recall 重視） | 1 スカラー合成 |
| modification_success_rate | 修正で target NG が消えた割合 |
| negative_transfer_rate | 修正で元になかった NG が新規発生した割合 |

3 seed × 層化分割（train 0.6 / val 0.2 / test 0.2）、temperature=0、モデルバージョン固定、すべて MLflow 記録。

### 3.4 フェーズ構成

| フェーズ | 内容 | ゲート |
| --- | --- | --- |
| Phase 0 | データ・ラベル監査 | 粒度・件数・クラス比の確認 |
| Phase 1 | 評価器 + 単一プロンプトのベースライン | T1 の床スコア確定 |
| Phase 2 | 再利用の対照実験（T2 reuse vs zerobase） | 再利用効果の実測 |
| Phase 3 | 頑健性と負の転移 | タスク記述言い換え、seed 変更で安定性確認 |
| クラウド確認 | Azure GPT-5.4 で同じ検証を反復 | ローカル結論の交絡確認 |

## 4. 結果: ローカル LLM vs クラウド強モデルの実測比較

### 4.1 Phase 1: T1 ベースライン（ローカル Qwen 2.5 14B Instruct）

プロンプト改良で v0.1.0 → v0.2.0 → v0.3.0 と反復:

| 反復 | TEST macro_recall | TEST macro_precision | TEST macro_F2 |
| --- | --- | --- | --- |
| v0.1.0 (素朴) | 0.648 ± 0.085 | 0.472 ± 0.083 | 0.580 ± 0.084 |
| v0.2.0 (欠落型明示) | 0.778 ± 0.056 | 0.478 ± 0.053 | 0.641 ± 0.046 |
| **v0.3.0 (厳格判定)** | **0.722 ± 0.056** | **0.533 ± 0.050** | **0.651 ± 0.047** |

v0.3.0 を Phase 1 確定ベースライン (P2) として採用。

### 4.2 Phase 2: reuse vs zerobase 対照実験（ローカル LLM、3 seed CI）

T2 修正タスクで reuse（T1 辞書を再利用）vs zerobase（辞書なし）を比較:

| variant | success_rate | 95% CI | negative_transfer |
| --- | --- | --- | --- |
| reuse | 0.612 ± 0.098 | [0.368, 0.855] | 0.769 ± 0.076 |
| zerobase | 0.400 ± 0.102 | [0.147, 0.652] | 0.520 ± 0.080 |
| paired diff | **+0.212** | - | +0.249 (95% CI [+0.015, +0.482]) |

per-pattern では 8/9 パターンで reuse 優位。特に明示型（jurisdiction +0.600、scope +0.467）で顕著。
**ローカル結果のみで判断すれば「再利用は強く成立」と結論できそう**。

### 4.3 落とし穴: 「負の転移」観測値の大半は評価器の False Positive

観測された負の転移率（reuse 0.769、paired diff +0.249）は統計的に有意で、これだけ見ると「再利用は負の転移を起こす」とも読める。

そこで **人手レビュー** を実施。reuse の new_ng_introduced=True サンプル 20 件を抽出し、
各「新規検出 NG」が **真の負の転移** か **T1 検出器の FP** か判定:

| 判定 | 件数 (27 件中) | 率 |
| --- | --- | --- |
| T (真の負の転移) | 0 | 0.0% |
| F (T1 検出器 FP) | 22 | 81.5% |
| ? (保留・元から残存) | 5 | 18.5% |

**真の負の転移率はゼロ**。観測値は T1 検出器 (P2 baseline の precision = 0.533) の FP がほとんど。

FP の主要発生源:
1. **条文範囲外への過剰検出** (例: 知財条項に派生資料未定義を判定)
2. **多段落規定の見落とし** (例: 同一条項の別段落に行政開示例外があっても欠落判定)
3. **「双方向 = 標準」の誤判定** (例: 「甲及び乙」双方向条項を不均衡判定)

### 4.4 失敗ルート: Few-shot プロンプト最適化と Structured Knowledge は 2 連続廃案

P4 (v0.4.0): few-shot で「対象外」例を 3 個追加。
P5-A (v0.5.0): 辞書に `applicable_topics` を追加して機械的トピック照合。

両方とも v0.3.0 を下回り廃案。

| プロンプト | macro_recall | macro_precision | macro_F2 |
| --- | --- | --- | --- |
| v0.3.0 (P2 確定) | 0.556 | 0.352 | 0.474 |
| v0.4.0 (P4 廃案) | 0.444 | 0.306 | 0.384 |
| v0.5.0 (P5-A 廃案) | 0.333 | 0.148 | 0.265 |

学び:
- **few-shot で「列挙する例」と「列挙しない例」を混ぜると判定が振れる**
- **機械的 hard filter は失敗 1 つで recall を連鎖的に失う**
- **プロンプトの局所最適化には限界がある。能力が交絡する**

### 4.5 Phase 3: タスク記述のパラフレーズ頑健性 (Prompt Robustness)

修正プロンプト v0.1.0（オリジナル）と v0.1.1（同意味・別表現）の比較:

| 指標 | v0.1.0 | v0.1.1 | 差 | 判定 |
| --- | --- | --- | --- | --- |
| reuse success | 0.550 | 0.455 | -0.095 | ±0.1 ぎりぎり OK |
| zerobase success | 0.289 | 0.289 | ±0.000 | 完全安定 |
| paired diff | +0.261 | +0.166 | -0.095 | 正方向維持 |

reuse 側は明示型 NG (jurisdiction -0.500, disclosure -0.400) で大きく揺らぐ。
辞書を参照する LLM の挙動がプロンプト文体に敏感。zerobase は変化なし。

### 4.6 クラウド強モデル確認: Azure OpenAI GPT-5.4 で同じ実験を反復（3 seed）

CLAUDE.md §3「クラウドの強モデルで最終確認」要件に従い、Azure OpenAI GPT-5.4 で再走。

| 指標 | ローカル qwen 14B | **Azure GPT-5.4** | 差 |
| --- | --- | --- | --- |
| reuse success | 0.612 ± 0.098 | **0.822 ± 0.022** | +0.210 |
| zerobase success | 0.400 ± 0.102 | **0.748 ± 0.090** | **+0.348** |
| **paired diff** | **+0.212** | **+0.074 ± 0.068** | **-0.138** |
| paired diff 95% CI | [+0.015, +0.482] | **[-0.095, +0.243]** | 0 を含む |
| per-seed paired (42/43/44) | 全 seed 正方向 | 0.000 / +0.133 / +0.089 | seed=42 で 0 |
| 所要 (3 seed) | 約 330 分 | 約 41 分 | -289 分 |
| 真の負の転移率（人手レビュー後） | ≒ 0 | ≒ 0 | - |

**強モデルでは zerobase が劇的に伸び (+0.348)、相対的な reuse 優位が約 65% 縮小**。
ただし完全消失ではなく、per-pattern では `nda_derivative_undefined`, `nda_disclosure_exception_missing` で +0.267 と意味的に細かい NG では reuse が有効。

### 4.7 1 seed だけで結論する罠（再現性とサンプル数の重要性）

Azure GPT-5.4 を seed=42 単発で見ると paired diff = **0.000**。
「強モデルでは再利用が完全消失」と早合点しそうになるが、3 seed 全体で見ると:

| seed | reuse | zerobase | paired diff |
| --- | --- | --- | --- |
| 42 | 0.844 | 0.844 | **0.000** （外れ値） |
| 43 | 0.800 | 0.667 | +0.133 |
| 44 | 0.822 | 0.733 | +0.089 |

**「単発の好スコアで合否判定しない」**（検証計画書 §5.4）が、クラウド検証でも適用された。
1 seed だけでは結論が反転するリスクがあることを実体験で確認。

### 4.8 ハイブリッド戦略の検証: 「クラウドで作った辞書をローカルで使う」

ここまでの観察を踏まえ、自然な仮説が出てきました:

> **「ローカル弱モデルの限界が『辞書の質』にあるなら、クラウドで精緻な辞書を作って弱モデルに与えれば、強モデル並みの性能が出るのでは？」**

これを Phase 4 として直接検証しました。

#### 設計

| 項目 | 値 |
| --- | --- |
| 辞書生成モデル | Azure OpenAI GPT-5.4 |
| 辞書のサイズ変化 | description 3 行 → **11 行** (約 3.6 倍精緻化)、examples 1 件 → 3 件 |
| 実行モデル | ローカル qwen 2.5 14B Instruct |
| 注意点 | ollama context size を 4096 → **8192** に拡張（カスタム Modelfile で対応） |

#### 結果

| 指標 | ローカル + 人手辞書 | **ローカル + クラウド辞書** | 差 |
| --- | --- | --- | --- |
| reuse success_rate | 0.550 | **0.548** | **-0.002** |
| zerobase success_rate | 0.289 | 0.364 | +0.075 |
| paired diff | +0.261 | +0.184 | -0.077 |
| reuse neg_transfer | 0.700 | 0.595 | -0.105 |

#### 重要な観察

**reuse success_rate は変化なし (-0.002)**。私が事前に「+0.65 - 0.75 程度」と推測した値とは大きく外れました。

つまり、**「クラウドで辞書を精緻化しても、ローカル弱モデルの T2 修正性能は伸びない」**。

#### なぜ伸びないのか

ローカル弱モデルにとっての限界は、辞書の質ではなく:

| ボトルネック | 内容 |
| --- | --- |
| 抽象表現の理解力 | GPT-5.4 が書いた精緻な法務表現を、qwen 14B が修正タスクに変換できない |
| 多段落の整合性チェック | 同一条項の別段落への参照判断が弱い |
| 法律文書の硬い文体生成 | reasoning モデル的な深い推論が必要な書き換え能力に天井 |
| コンテキスト使用効率 | 辞書が冗長になると注意配分が薄まる |

要するに **「教科書を高度にしても、生徒の能力が上がらない」**。

#### zerobase が +0.075 改善した「見かけ」

注意: zerobase は辞書を使わないのに success が向上しています。これは **修正自体の改善ではなく、辞書 v0.4.0 を使う T1 検出器の判定変化** による見かけの効果と解釈できます。
- zerobase の修正プロンプトには辞書は入らない
- ただし「target NG が修正後に検出されない」を測る検出器が辞書 v0.4.0 を使う
- 検出器の判定が変わったため、success_rate の見かけが動いた

reuse の neg_transfer 改善 (-0.105) も同様で、**辞書を精緻化することで T1 検出器の precision（特に条文範囲外検出 FP）が下がった** と解釈できます。

#### Phase 4 の結論

| 用途 | 価値 |
| --- | --- |
| ローカル弱モデル運用での **T2 修正性能向上** | **価値なし**（reuse +0.000） |
| ローカル弱モデル運用での **T1 検出精度向上** | あり（neg_transfer -0.105） |
| 「弱モデルの天井を辞書で破る」 | **不可** |

**モデル能力 > 辞書品質** という非対称性が、本検証で繰り返し観測されたパターンです。

## 5. 結論: 知識層再利用は「モデル能力依存で条件付き支持」

### 5.1 仮説に対する判定（4 段階で評価）

| 評価軸 | 結果 |
| --- | --- |
| 弱モデル運用での再利用優位 | **強く支持** (paired diff +0.212, 全 seed 正方向, CI 0 を含まず) |
| 強モデル運用での再利用優位 | **弱く支持** (paired diff +0.074, 2/3 seed 正方向, CI 0 を含む) |
| 真の負の転移なし | **両モデルで支持** (人手レビュー後 ≒ 0) |
| **クラウド辞書 × ローカル実行で性能改善** | **不支持** (reuse +0.000、辞書品質では弱モデルの天井を破れない) |
| モデル能力依存 | **強い**（強モデルでは zerobase が +0.555 向上、ハイブリッドでは reuse 不変） |

### 5.2 実装者への含意: 投資先の判断マトリクス

| 運用シナリオ | 知識層再利用の価値 |
| --- | --- |
| ローカル小型モデル（コスト効率優先） | **必須**。zerobase が 0.289 で壊滅、辞書投入で 0.612 まで救済 |
| クラウド強モデル（最高性能優先） | **補助的**。zerobase で 0.748 達成、辞書投入は特定パターンのみ +0.267 上積み |
| ハイブリッド（クラウド辞書 × ローカル実行） | **NO**。辞書を精緻化しても reuse は +0.000、ローカルの天井はモデル能力にある |
| 意味的に細かい NG (derivative / disclosure) のみ強モデル | **YES**。強モデルでも +0.267 効く |
| 教育・説明可能性 | **常に有用**。「何が NG か」を人間と共有する仕組みとして |

### 5.3 一文で言うと

> **知識層再利用は弱モデルの能力補完として強く効く (+0.212)。
> 強モデルでは zerobase 自身が高性能になるため相対的価値は縮小 (+0.074)。
> 辞書品質を上げてもローカル弱モデルの天井は破れない (+0.000)。
> 「モデル能力 > 辞書品質」 という非対称性が、本検証で繰り返し観測された。**

## 6. Agentic AI 検証から得た 4 つの実装知見

### 6.1 「単発の好スコアで判定しない」を実体験（評価駆動開発の基本）

ローカル 1 seed では reuse 優位 (+0.261)、Azure GPT-5.4 seed=42 単発では消失 (0.000)。3 seed CI で初めて全体像が見える。
LLM-as-judge や single-shot ベンチマークでも同じ罠が起きうる。

### 6.2 LLM-as-judge の補正に人手レビューを噛ませる

T1 検出器の precision = 0.533 で、観測される negative_transfer_rate の大半は False Positive。
人手レビューで「真の値はゼロ」と判明したことが、結論を「仮説不支持」から「条件付き支持」に書き換えた。
**評価器自体の品質確認は必須**。

### 6.3 失敗ルートを記録する（Few-shot / Structured Knowledge）

T1 改良で P4 (Few-shot 3 例) / P5-A (Applicable Topics 機械照合) と 2 連続廃案。
**Few-shot で「列挙する例」と「列挙しない例」を混ぜると判定が振れる**、
**機械的 hard filter は失敗 1 つで recall を連鎖的に失う**。
うまくいかなかった理由を残すことが、次の試行への最大の学習。

### 6.4 プロバイダ非依存の LLM 設定層を最初から作る

ローカル ollama → Azure OpenAI へ `.env` の値だけで切替できる構造にしておいたおかげで、クラウド確認が 1 日で完了。
OpenAI SDK の AzureOpenAI クライアント分岐、reasoning モデル（GPT-5 / o1 / o3 系）の `max_completion_tokens` 仕様吸収など、**最初から疎結合に設計する価値**。

## 7. やって失敗したこと（次回の戒め）

| 試行 | 失敗の根本原因 |
| --- | --- |
| T1 検出器 v0.4.0 (few-shot 3 例) | 明示型 NG の recall を犠牲にして FP 削減を狙ったが、recall -0.112 で F2 大幅低下 |
| T1 検出器 v0.5.0 (機械的トピック照合) | LLM の主題判定ミスで recall を連鎖的に失う。F2 -0.209 |
| 1 seed だけで結論しようとした Azure seed=42 | 偶然 paired diff 0.000 で「再利用消失」と早合点しかけた |
| **ハイブリッド (クラウド辞書 × ローカル)** | **「辞書を精緻にすれば弱モデルでも強モデル並み」と期待したが reuse +0.000。天井はモデル能力にあった** |

## 8. 制約と今後の課題（別ドメイン / ハイブリッド運用 / 法務妥当性）

| 制約 | 内容 |
| --- | --- |
| ドメイン依存性 | NDA のみで確認。業務委託契約、ライセンス契約、利用規約等への横展開は未確認 |
| サンプル数 | 1 seed あたり 45 サンプル。3 seed で 95% CI 幅は依然広い |
| T1 検出器の天井 | v0.3.0 で macro_F2 0.651 が床。さらに上はモデル能力で稼ぐしかない |
| reasoning モデルの再現性 | GPT-5.4 は temperature 1.0 固定、seed 不可。完全再現は不可 |
| 評価指標の偏り | NG Recall 主軸。修正テキストの「法務的に妥当か」は別評価器が必要 |

今後の課題:
- 別ドメイン (業務委託、ライセンス等) で同じ検証を反復
- T1 検出器を `applicable_topics` + few-shot 「対象外例のみ」で再設計
- 修正テキストの法務妥当性を人手 / LLM-as-judge で別評価
- ハイブリッド運用 (弱モデルで T1 検出、強モデルで T2 修正) のコスト/性能パレート

## エピローグ: 「検証ループを回さないと見えない」もの

「やってみないと分からない」ことは多いが、本検証で見えたのは:

1. **ローカル弱モデルでの「驚くべき結果」(+0.212) は、強モデル基準では妥当な改善幅 (+0.074)**。
2. **「負の転移が観測された」は評価器側の FP が混入していることが多い**。人手レビューで補正必須。
3. **モデル能力交絡を切り分けるには、ローカル + クラウド両方で同じ実験を反復するしかない**。
4. **「単発の好スコア」「単発の悪スコア」のいずれも、3 seed CI を取らないと判断できない**。
5. **「クラウドで辞書を作ればローカルでも強モデル並み」は淡い期待で、ハイブリッドは reuse +0.000**。モデル能力 > 辞書品質。

これらは個別には「教科書通り」だが、実際に検証ループを回して数字で殴られると重みが違う。
**Agentic AI / LLM エージェント自動生成** に取り組む方の参考になれば幸いです。
