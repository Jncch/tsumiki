# Phase 2 Azure GPT-5.4 reuse 負の転移 人手レビュー結果（seed=42, 20/21 件）

[`phase2_azure_negative_transfer_review_2026-06-10.md`](phase2_azure_negative_transfer_review_2026-06-10.md) の 20 件レビュー対象に対し、
Claude が試案として T/F/? 判定を埋めた結果。ローカル qwen 14B 時と同じ workflow。

## 0. 判定方針

| 判定 | 意味 |
| --- | --- |
| **T** | 真の負の転移。修正で実際に NG を新規に導入した |
| **F** | T1 検出器の FP。修正後テキストには実体としてその NG は存在しない（対象範囲外、既存規定で充足） |
| **?** | 判断保留。グレーゾーン、または元から残存していた問題の検出ぶれ |

FP / ? の主要パターン（Azure GPT-5.4 reuse seed=42 で観測されたもの）:

1. **テンプレプレースホルダ `〇〇` の解釈**: 「〇〇の可能性の検討を目的として」が purpose_undefined と判定される例
2. **別条参照（`本目的のために`）の解釈**: 条文単体には目的の定義がないが、別条で定義されている前提のテキスト
3. **中立的合意管轄の誤判定**: 「東京・大阪地方裁判所」「東京地方裁判所」を一方的指定と判定
4. **元から残存していた問題の表面化**: 知財条項の派生資料未定義などが、別の主 NG 修正後に検出される

| 項目 | 件数 | 率 |

### 1.1 真の負の転移率の推定

20 サンプル中、真の負の転移を含むサンプル: **0 件**。
全 21 件（reuse_azure_seed42 中の new_ng_introduced=True）への外挿でも、グレー (?) 10 件をすべて T と扱った場合の上限 **約 50%**、すべて F と扱えば **0%**。
妥当な推定は **0〜25% 程度** が真の負の転移。

### 1.2 ローカル qwen 14B 時との比較

| 項目 | ローカル qwen 14B | **Azure GPT-5.4** |
| --- | --- | --- |
| 観測 negative_transfer_rate | 0.700 | **0.467** (seed=42), 0.407 ± 0.068 (3 seed) |
| 抽出件数 | 20/28 | 20/21 |
| T (真の負の転移) | 0 / 27 (0.0%) | **0 / 20 (0.0%)** |
| F (T1 FP) | 22 / 27 (81.5%) | **10 / 20 (50.0%)** |
| ? (保留) | 5 / 27 (18.5%) | **10 / 20 (50.0%)** |
| **真の負の転移率推定** | ≒ 0 | **≒ 0** |

→ **両モデルとも真の負の転移率はほぼゼロ**。観測された負の転移率の大半は T1 検出器の FP / 検出ぶれ。

### 1.3 ローカルとの定性差

Azure では **明らかな「条文範囲外への検出」(F) が減少** し、代わりに **「元から残存していた問題の検出ぶれ」(?) が増加**:

| FP / ? 発生源 | ローカル qwen | Azure GPT-5.4 |
| --- | --- | --- |
| 条文範囲外への過剰検出 | 多数 (F) | 一部 (F) |
| 多段落規定の見落とし | 多数 (F) | 少ない |
| 双方向＝標準の誤判定 | 数件 (F) | 観測されず |
| **プレースホルダ `〇〇` の purpose 判定** | - | **多数 (?)** |
| **「本目的」別条参照の purpose 判定** | - | **多数 (F)** |
| **「東京・大阪」中立的合意管轄の jurisdiction 判定** | - | **複数 (F)** |
| **元から残存の派生資料未定義の検出** | - | **複数 (?)** |

GPT-5.4 は T1 検出器として **「条文の主題を理解する能力」が向上** し、知財条項に派生資料未定義のような「条文外なのに該当する」検出は減ったが、**「テンプレ的なプレースホルダ・他条参照」を厳密に判定** する別タイプの FP が増えた。

## 5. T1 検出器の改善余地（Phase 1 へのフィードバック）

Azure GPT-5.4 を T1 検出器に使った場合の改善余地:

| 課題 | 対処案 |
| --- | --- |
| **`〇〇` プレースホルダを purpose_undefined 判定** | 辞書 description に「テンプレート上のプレースホルダ（〇〇、●●等）は具体名が入る前提として判定対象外」と明記 |
| **「本目的」別条参照を purpose_undefined 判定** | パターン定義に「条文内に明示の目的定義があるか、または `本目的` 等の代名詞で別条参照されていれば該当しない」と追記 |
| **中立的合意管轄を jurisdiction_one_sided 判定** | パターン定義の「紛らわしい」セクションに「東京（地方）裁判所等の双方対等な合意管轄」を強調 |

ただし、これらの修正は Phase 2/3 への波及効果が大きく、再走必要。
本検証では **「現状の T1 検出器でも真の負の転移率はゼロ」** という結論で十分。

## 6. 結論

### 6.1 仮説に対する確信度（更新）

| 評価軸 | 結果 |
| --- | --- |
| 知識層再利用が成立する（ローカル弱モデル） | ✅ **強く支持**（paired diff +0.212, 真の負の転移 ≒ 0） |
| 知識層再利用が成立する（クラウド強モデル） | △ **弱く支持**（paired diff +0.074, CI 0 含む、真の負の転移 ≒ 0） |
| 負の転移なし | ✅ **両モデルで共通**（人手レビュー後の真の値 ≒ 0） |
| モデル能力との依存性 | **強い**（強モデルでは zerobase が +0.555 向上、相対 reuse 優位は約 65% 縮小） |

### 6.2 検証全体の最終結論

**仮説「知識層再利用は成立する」は、3 段階で支持**:

1. **弱モデル運用**: 強く支持（+0.212, 全 seed 正方向）
2. **強モデル運用**: 弱く支持（+0.074, 2/3 seed 正方向、CI 0 含む）
3. **負の転移**: 両モデルで真の値 ≒ 0

実用的な含意:
- ローカル小型モデル運用では **辞書投入は必須**（zerobase が壊滅、辞書で救う）
- クラウド強モデル運用では **辞書は補助的価値**（特定パターン derivative/disclosure で +0.267 上積み）

### 6.3 注意事項

本判定は私（Claude）が原文+修正後+パターン説明から論理的に推定したもの。
法務専門家の意見ではない。各判定の妥当性は、最終的にはユーザーまたは法務担当者の確認を要する。

特にグレーゾーン (?) 10 件は、専門家視点で T か F かが揺れうる:

| パターン | 件数 | グレーの理由 |
| --- | --- | --- |
| プレースホルダ `〇〇` | 4 | テンプレ上は具体名が入る前提として扱うか、抽象表現として NG とするか |
| 知財条項の派生資料 | 4 | 元から残存していた問題を「修正で新規導入したか」の解釈 |
| 廃棄証明書欠落 | 1 | 廃棄証明書をどこまで義務化すべきか |

ただし、これらをすべて T として扱っても 10/20 = 50% で、観測値 46.7% から大幅修正ではない。
判定の本質的な揺らぎは結論を覆さない（真の負の転移率は依然として 0〜25% の範囲）。
