# Phase 5c 結果: 目的駆動 E2E 試走 (seed=42)

設計: [phase5c_design.md](phase5c_design.md)

## 0. 結論

> 自然言語の目的「NDA をレビューして NG 条項を直したい」を入力として、
> 目的駆動パイプライン全体 (parser → lookup → eval 動的ロード → knowledge auto → policy compose 相当 → evaluator) が
> 一気通貫で動作した。
> **paired diff = +0.261 で Phase 5a V0 / Phase 5b と完全一致**し、ゲート通過 (|diff|=0.000)。
> 流用蓄積から既存評価器 `modification_success_v1` が exact_match で自動選択され、generator は呼ばれず、
> 設計の「目的駆動セットアップ」「シナリオ非依存基準」を NDA で実証した。

## 4. 実装上の発見と修正

| 発見 | 修正 | 場所 |
| --- | --- | --- |
| parser の ChatFn 型不整合（`ChatResult` vs `str`） | `_to_text_chat_fn` アダプタを追加し、`.content` を取り出す | `src/tsumiki/runner/e2e.py` |
| qwen 14B が「レビューして直す」を `modify` と判定 | parser プロンプト v1 に「複合動作は `detect_and_modify` を選ぶ」を追記 | `src/tsumiki/goal/parser.py` |
| parser が `outputs` を 1 つだけ返すため signature 不一致 | parser プロンプト v1 に `detect_and_modify` の標準 outputs（findings + modified_document）を明示 | 同上 |
| 試走中に新規生成された不要な評価器 (`nda_modification_success_v1`) | 流用パスに乗せるため削除して再実行 | `eval/generated/nda/detect_and_modify/` |

これらの修正で qwen 14B でも一気通貫が安定動作するようになった。第三者再現にはクラウド強モデルでの追加確認が推奨 (CLAUDE.md §3)。

## 7. Phase 6 への申し送り

- 目的駆動パイプラインは NDA で動作実証。Phase 6 は ISO27001 ドメインを起点に同じパイプラインが動くかを検証する
- ISO27001 では evaluator が未蓄積のため、初回実行で **generator パス** が動く。これが Phase 5c で未実施のテストで、Phase 6 の合格条件に組み込む
- Knowledge も初回は extract が必要。`knowledge/extractors/` 実装は Phase 6 範囲
- 同一 `examples/` ディレクトリ構成で nda と iso27001 が共存できることが Phase 6 の合格条件
- ISO27001 の `task_class` は `detect_and_modify`（不備検出と是正案）または `detect`（チェックリスト検出のみ）で運用試験
