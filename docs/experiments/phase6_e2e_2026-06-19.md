設計: [phase6_design.md](phase6_design.md)

## 0. 結論

> ISO27001 ドメインで目的駆動パイプライン全体が動作し、**reuse paired diff = +0.029 で正の寄与を観測**。
> 設計 §4.2 の合格条件「paired diff > 0」を満たし、N=2 ドメインで知識層再利用効果を再現した。
> ただし NDA の +0.261 と比較すると効果は **約 1/9** に縮小しており、**ドメインによる効きの差** が観測された。
> 同一 `runner/e2e.py` で NDA と ISO27001 が動くこと、Knowledge schema (Phase 5a 確定) が ISO27001 でも
> 情報損失なく流用できることを実証した。51 件の unit test がリグレッションなしで通過。
> generator パスは qwen 14B では品質不足を示したため流用パスで進めた。

## 3. 効果の縮小に関する解釈

NDA で +0.261 だった効果が ISO27001 では +0.029 に縮んだ。考えられる要因と相対的な重みを整理する。

| 仮説 | 根拠 | 重み |
| --- | --- | --- |
| (a) ドメイン特性: ISO27001 のような技術的・規範的なドメインでは qwen 14B 自体が一定の知識を持つため zerobase が高い (0.381 vs NDA 0.289) | zerobase の絶対値が NDA より +0.092 高い。Phase 4 §4.3 の「強モデルでは zerobase 自身が高性能になるため相対的価値は縮小」と同型の現象 | **大** |
| (b) Knowledge スキルの description が ISO27001 で最適化されていない | Phase 5a で NDA 用 description は ablation で寄与を測ったが、ISO27001 では未実施 | 中 |
| (c) clean clauses 10 件はサンプル数として少ない可能性 | NDA も 10 件で運用しており構造は同じ | 小 |
| (d) synth 3 skip / modify 3 skip でサンプル損失 | サンプル数差は微小 (45 -> 42 -> 39) | 小 |

最も支配的なのは (a) のドメイン特性。Knowledge 層再利用の効きは「ベースラインの zerobase がどれだけ低いか」に依存する、というのが Phase 1〜4 + Phase 6 の通底する発見。

## 4. データの限界 (CLAUDE.md §11 「忖度せず」「わからないことは『わからない』」)

- ISO27001 の clean clauses 10 件は LLM 合成ではなく手書きで作成。**業務データに依存しない一方、実運用文書との分布乖離が残る可能性**。第三者の追試では実規程テンプレートとの比較が必要
- 9 不備パターンは Annex A の主要 4 領域からサンプリングしたが、選定基準は「NDA と件数を揃える」「カテゴリ多様性を確保」という設計判断であり、ISO27001 監査の頻出度に基づくものではない
- seed=42 のみ。複数 seed で CI を取れていない。Phase 2 同等の 3 seed CI 取得は Phase 7 以降の課題
- 評価は決定関数による `modification_success_rate` のみ。「是正案が ISO27001 監査として実務的に妥当か」の品質評価は LLM judge + 人手較正が必要（Phase 5c ガードレール雛形を流用予定）

## 5. generator パスの結果と影響

設計 §4.3 で「(b) 流用不可なら generator が新規生成し verify を通過して保存」を合格条件のうち選択肢として用意していた。今回の実行では:

1. `experiments/test_phase6_generator.py` で qwen 14B 単体テストを実施
2. JSON 構造としては有効な EvaluatorSpec を生成したが、`implementation` の中身が意味的に壊れていた (`target_pattern_1` を文字列リテラルとして埋め込む等)
3. `verifier` が test_cases の expected と乖離を検出し、unit test 機構として正しく失敗を返した
4. 流用パスに切り替えるため `experiments/seed_iso27001_evaluator.py` で seed 評価器を投入し、Phase 6 試走を流用パスで完了させた

### 結論

- `goal/generator.py` + `verifier.py` の実装は正しく動作している (JSON 仕様遵守、test_cases による安全網)
- qwen 14B では生成された Python コードの **意味的品質** が不足
- これは CLAUDE.md §3「自動生成・探索の品質はクラウドで最終確認」と整合する観測
- generator パスの実用性検証は Phase 7 以降にクラウド GPT-5.4 で別途実施する
