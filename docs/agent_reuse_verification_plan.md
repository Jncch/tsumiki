# タスク実行エージェント自動生成における「知識層再利用」検証 方針・計画

## 0. このドキュメントの位置づけ

本書は、これまでの検討で確定した方針と、最小検証の設計、参考にできる既存OSSと評価指標を整理したものである。
結論を先に置く構成とし、未確定事項は末尾にまとめる。表現は検討時のものを保持し、勝手な言い換えは行わない。

---

## 1. 結論サマリ

- 作るのは「汎用の自律エージェント自動生成フレームワーク（ゼロベース）」ではない。これは既存研究の再発明であり非効率なため不採用。
- 中核の価値は「自動生成」そのものではなく「生成された部品の再利用（転用）」に置く。
- 再利用が成立する粒度は明確に分ける。再利用できるのは知識層とツール層であり、最適化済みポリシー層はタスクごとに作り直し＋再評価が必要。
- この仮説の一般原理は既に実証済み（スキルライブラリ系研究）。未検証なのは「業務文書・法務のような縦ドメインで、転用関係にある2タスク間に効くか」という具体形。
- したがって本検証の正味の価値は「既知の原理を未検証のドメインで確認し、負の転移の有無と工数削減効果を実測する」ことにある。

---

## 2. 方針（採用・不採用の決定事項）

| 項目 | 決定 | 理由 |
| --- | --- | --- |
| 汎用フレームワークの自作 | 不採用 | AFlow / ADAS / AgentSquare / A2Flow 等で飽和。限界貢献が小さい |
| 既存OSSの fork + 差分貢献 | 採用 | 再現性と比較可能性を保てる |
| 自律エージェント群のオーケストレーション自動生成 | 当面の対象外 | 探索空間が広く不安定・評価困難。後段の研究課題に切り出す |
| 制約されたマルチエージェント（役割ノードの合成まで） | 採用 | 既存探索手法の射程内で、必要に応じてマルチ化される |
| 差別化の置き場所 | アルゴリズムではなく「整備されたドメイン縦資産」 | 知識・ツール層が転用の実体だから |

---

## 3. 中核仮説と三層分離

「ドメイン知識領域は再利用可能」を1語で扱わず、再利用粒度ごとに3層へ分解する。

| 層 | 内容 | 再利用 | 再評価の要否 |
| --- | --- | --- | --- |
| 静的知識資産 | 法令、条項タクソノミー、ポリシー文書、過去事例。RAGコーパス／ナレッジベース | 可（タスク非依存） | 不要（鮮度管理は別途必要） |
| ツール／インターフェース | 条項検索、管轄ルール照合、引用チェック、フォーマッタ等の決定的関数群 | 可（入出力契約が同じ範囲） | 不要（単体テストで担保） |
| ドメイン特化ポリシー | 役割定義、観点リスト、最適化済みプロンプト・ノード構成 | 半再利用（種は流用、最適化はやり直し） | 必要（タスク目的が変わると劣化） |

設計上の含意:

- バージョン管理の境界を分ける。知識・ツール層はドメイン資産として長期管理。ポリシー層はタスクごとの使い捨て成果物として管理。両者を混ぜない。
- 評価ループに乗せるのはポリシー層の振る舞いのみ。知識・ツール層は単体テスト（関数の正しさ、検索の再現率）で担保しタスク評価から外す。
- 正味の工数削減は「知識収集・ツール実装の再投資」分のみ。ポリシー設計・チューニング・評価は毎回かかる。知識整備が重いドメインほど旨味が大きい。

## 4. 既知と未検証の切り分け

### 4.1 既に実証されている（一般原理）

- スキルライブラリ系の研究系列（Voyager を起点）が、知識・スキル層がタスクを越え、さらにエージェントやモデルを越えて転移し得ることを実証している。
- 「能力（知識・スキル）をモデル重みから切り離す」という設計思想が共通前提になっている。
- 同時に、最適化済みプロンプト（ポリシー）はモデルやタスク分布のずれで劣化するという知見もあり、ポリシー層の作り直しが必要という主張を裏付ける。

### 4.2 未検証（本検証の対象）

- 既存ベンチマークはゲーム、web操作、embodied、QA、金融推論などに偏る。
- 業務文書・法務の縦ドメインで、転用関係にある2タスク間（例: チェックと作成）に三層分離が効くかは確認できていない。
- 留意: 関連研究の多くは2026年の新しいプレプリントで査読前を含む。「強いが若い」証拠として扱う。

## 5. 検証設計

### 5.1 前提原則（スレッド全体の合意事項）

- 評価器が先、自動生成は後。探索や再合成は最大化するスカラースコアが無ければ起動しない。
- 評価条件は実験前に固定する（後出ししない）。
- 再利用は評価を免除しない。再合成しても新タスクの評価信号で検証し直す。

### 5.2 フェーズ構成

| フェーズ | 内容 | ゲート |
| --- | --- | --- |
| Phase 0 データ・ラベル監査 | 対象データの粒度（文書単位か条項単位か）、件数、クラス比、ラベル付与基準の一貫性を確認 | 粒度次第でポリシー最適化ループの可否が決まる |
| Phase 1 評価器＋ベースライン | 主指標を計算する評価関数を実装。単一プロンプトのベースラインを作る。test分割を層化固定、temperature=0、モデルバージョン固定、全試行をMLflowに記録 | ベースラインのスコア・コスト・レイテンシが以降の比較の床 |
| Phase 2 再利用の対照実験 | T2をゼロベースで構築 vs T1の知識・ツール層を注入して構築 | 知識層再利用の効果を実測 |
| Phase 3 頑健性・負の転移 | タスク記述の言い換え・シード変更で再実行し安定性を確認。再利用層が新タスクの足を引っ張らないか確認 | 不安定なら再現性の観点で否定的結論 |

### 5.3 対照実験の核

- 設定: 1ドメイン、転用関係にある2タスク（T1, T2）。
- 比較: 「T2 ゼロベース構築」対「T2 = T1の知識・ツール層を注入して構築」。
- 測るもの: 目標到達までの工数・反復回数（コールドスタート差）、最終スコア、負の転移の有無と大きさ。

### 5.4 合格・判断ルール（実験前に固定）

- 知識層再利用がゼロベースに対しコールドスタート工数を有意に削減し、最終スコアを劣化させず、負の転移が出ない → 仮説確認。縦1本で前進。
- ゼロベースと差が無い、または負の転移が出る → そのドメイン・タスク対では再利用前提が成立しない、という明確な結論。

## 7. 評価指標まとめ

### 7.1 ラベルが定義できるタスク（分類・チェック系）

| 指標 | 用途 | 備考 |
| --- | --- | --- |
| NG Recall（見逃し率の裏返し） | 主指標 | 業務上の致命傷はNGをOK判定する見逃しのため |
| Precision | 補助 | 全件NG判定への偏りを抑える |
| F1 / F-beta（beta>1） | 補助・合成 | Recall重視で1スカラーに合成する場合 |
| 混同行列 | 補助 | 誤りの内訳確認 |
| Accuracy | 参考のみ | クラス不均衡下では単独使用しない |
| Latency | コスト面 | 構成比較に必須 |
| Cost（USD） | コスト面 | 探索・実行のトークン費用 |

多目的化する場合は、コスト・レイテンシ・頑健性をペナルティ項として1スカラーに合成する（ADASが未解決と挙げた multi-objective 方向）。意味の薄い項目の追加はノイズになるため避ける。

### 7.2 再利用の対照実験

| 指標 | 用途 | 由来 |
| --- | --- | --- |
| 再利用率（reuse rate） | 知識・ツール層がどれだけ再使用されたか | Skill-Pro 等 |
| クロスタスク成功差（cross-task success delta） | ゼロベース対再利用の最終性能差 | スキル再利用系 |
| コールドスタート工数（目標到達までの反復回数） | 立ち上げ工数の削減効果 | 本検証の主目的 |
| 負の転移（negative transfer）の有無・大きさ | 再利用層が新タスクの足を引っ張らないか | 本検証のリスク管理 |
| 冗長性（redundancy） | ライブラリ肥大の確認 | SkillGraph |

### 7.3 開放タスク（アイデア発想など。今回の主土俵ではないが参考）

| 指標・手法 | 用途 | 注意 |
| --- | --- | --- |
| ペアワイズ / ELO（Idea Arena型） | 絶対採点より人間選好に整合 | 単一スカラーへの落とし込みに一手間 |
| 複数批評パネル（3体以上） | 単一判定器の偏り低減 | ハッキングは完全には防げない |
| 埋め込みベースの多様性・新規性 | 1案収束を防ぐガードレール | 単独では不十分 |
| 人手較正（判定器と人間の一致度） | 判定器の妥当性確認 | 一致度が低ければループ無効 |

開放タスクでは、生のLLM判定スカラーへ直接最適化しない。報酬ハッキング（見栄えだけ良く中身の空虚な出力への収束）を招くため。

## 8. リスクと留意点

- 知識層の再利用は「正しさの再利用」ではない。法令・ポリシーは改定されるため、知識資産に鮮度管理（出典・更新日・有効性）が必須。古い知識の再利用はポリシーが正しくても誤出力を生む。
- ツール層の流用は入出力契約が同じ範囲でのみ安全。タスク変更で必要な入出力が変われば改修対象。
- 自動設計手法全般に頑健性の弱さ（タスク記述の言い換えで結果がぶれる）が報告されている。再現性最優先の観点で、合否は単発の好スコアで決めない。
- 探索中のコストは大きい。検証時は validation セットを小さく絞る。
- AFlowはOperatorに移行起因のバグが残る可能性があり、デバッグ工数を見込む。

---

## 9. 次に確定すべき事項

1. 対象ドメインの確定（知識整備が重い領域ほど方式の旨味が出る）。
2. そのドメイン内で転用関係にある2タスク T1, T2 の確定。
3. T1のデータ・ラベルの粒度（文書単位か条項単位か）、件数、クラス比の確認（Phase 0）。
4. 主指標を計算する評価関数の定義（ラベル系なら NG Recall を主に F-beta 等）。

上記が決まれば、対照実験の設計はそのまま固定できる。
